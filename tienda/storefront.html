Resolución de Identidad

¿A qué tienda
pertenece
esta visita?

getStoreRef() resuelve en cascada: primero mira route.params.storeRef, y si no hay, compara el hostname actual con los dominios internos. Si no coincide con ninguno, asume que es un dominio personalizado y lo usa como referencia.

route.params.storeRef

El caso normal. La ruta pública es /store/:storeRef con el slug o ID de la tienda.

Dominio personalizado

Si el hostname no es localhost, la IP local ni el dominio principal, se usa como storeRef directamente — la tienda responde en su propio dominio.

Redirect de slug canónico

Si la API devuelve un slug diferente al usado en la URL (p.ej. se accedió por ID), hace router.replace silencioso hacia el slug definitivo para consolidar el canonical.

Vista por defecto desde la config

Si storefront_config.default_view === 'services', la tienda arranca en la agenda sin que el cliente tenga que cambiar la URL.

URL estándar (slug)

aromas-del-sur.com/ (hostname como storeRef)
aliases de view soportados ?view=services ?view=appointments ?view=classes
Los tres apuntan a la vista de servicios. Los alias existen por compatibilidad con links externos.

Switch y Reactividad

Dos vistas.
Un componente.

El switch entre tienda y servicios es simple: currentView controla cuál de los dos componentes se monta con v-if. Tres watchers mantienen todo en sincronía cuando la URL cambia desde fuera.

Cliente toca "Servicios" en el toolbar
changeStoreView('services')
Desmonta
StoreCatalog

v-if queda false

Monta
StoreAppointments

v-if queda true

URL actualizada a ?view=services con router.replace
Los tres watchers
route.params.storeRef
Cuando el usuario navega a otra tienda sin recargar, relanza loadProfile() completo.
route.query.view
Actualiza currentView si alguien pega un link con ?view=services o si el botón de atrás lleva a otra vista.
catalogSeoProducts
Reactiva updateSeo() cuando el catálogo hijo termina de cargar y emite sus productos, enriqueciendo el JSON-LD con el ItemList.
