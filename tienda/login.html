Coordinación entre Modales

Dos modales.
Un overlay.
Cero fricciones.

El componente padre (generalmente el storefront) monta ambos modales y escucha sus eventos. El switch entre Login y Register no cierra el backdrop — simplemente alterna qué modal está visible con un v-if.

Cada modal emite exactamente lo que necesita el padre para orquestar la transición. No hay estado compartido entre ambos.

@close @success @switch-to-register @switch-to-login @open-orders
StoreLoginModal visible

El cliente ve el formulario de login. Si hace submit correcto, se llama a auth.login() y se emite @success con el usuario.

emit('success', auth.user)
Switch → Register

El link "Registrate gratis" emite @switch-to-register. El padre oculta el login y muestra el registro sin cerrar el overlay.

emit('switch-to-register')
Registro exitoso → pantalla de verificación

El modal de registro reemplaza el formulario con la pantalla de éxito. El formulario desaparece, aparece el bloque con instrucciones y el botón de reenvío.

successMsg.value = result.message
Switch → Login desde verificación

El botón "Ya verifiqué mi correo" emite @switch-to-login. El padre vuelve a mostrar el login.

emit('switch-to-login')
Login exitoso → cierre

@success llega al padre con el usuario autenticado. El padre cierra el overlay y actualiza el estado de sesión en el storefront.

Calidad del Email

El sistema
cuida la calidad
del dato.

Antes de llegar al servidor, el email pasa por dos utilidades del módulo emailGuards. Una normaliza el string para evitar duplicados por espacios o mayúsculas. La otra detecta dominios sospechosos y muestra un hint al usuario sin bloquear el registro.

normalizeEmail()

Hace .trim().toLowerCase() antes de enviar al servidor y antes de comparar email con confirmEmail. Evita que "Nacho@Email.com " y "nacho@email.com" se registren dos veces.

normalizeEmail(" NACHO@Email.com ") → "nacho@email.com"
getSuspiciousEmailHint()

Detecta dominios descartables o temporales (ej: mailinator.com, yopmail.com). Muestra un hint naranja debajo del campo sin impedir el registro. Es una advertencia, no un bloqueo.

getSuspiciousEmailHint("a@yopmail.com") → "Este dominio parece..."
Validación de confirmación

Email y confirmEmail se comparan después de normalizados. Lo mismo para la contraseña. Si no coinciden, el error aparece antes de llamar al servidor.

Verificación de Correo

No se entra
sin verificar
el email.

El registro no da acceso directo. El sistema manda un correo con un enlace de verificación. Si el cliente intenta loguear antes de verificar, el login devuelve un error específico que activa el botón de reenvío.

El botón de reenvío solo aparece si el mensaje de error contiene la frase "verificar tu correo". Es una detección por string, simple y efectiva.

1
Registro exitoso

El formulario desaparece. La pantalla de éxito muestra el mensaje del servidor y el email registrado. Aparecen dos botones: "Reenviar" y "Ya verifiqué".

2
Reenvío del correo

Llama a api.resendVerification({ email }) con el email guardado en registeredEmail. Actualiza el mensaje de éxito con la respuesta del servidor.

3
Intento de login sin verificar

El store devuelve un error con la frase "verificar tu correo". El computed canResendVerification lo detecta y muestra el link de reenvío también en el login.

4
Login exitoso post-verificación

auth.login() resuelve, se emite @success. El overlay se cierra y el cliente queda autenticado en el storefront.

Diseño Adaptable

El botón adopta
el color de cada tienda.

Todos los botones de acción usan var(--theme-primary). El storefront inyecta esa variable en el overlay y los modales se repintan automáticamente. Sin props de color, sin lógica extra.
