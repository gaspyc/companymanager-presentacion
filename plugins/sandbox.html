Para Desarrolladores

Sandboxing
Seguro

Un plugin puede hacer cosas increíbles, pero nunca puede romper el sistema. El sandboxing es el contrato técnico que lo garantiza para todos los comercios.

Reglas claras, límites bien definidos y una capa de aislamiento que protege tanto al comercio como al desarrollador.

Sin acceso a BD Permisos declarados Ejecución aislada Auditoría humana

La Regla #1

Cero Acceso Directo a la Base de Datos

Un plugin no puede abrir una conexión directa a la base de datos, leer tablas libremente ni ejecutar queries propios. Nunca.

Todo lo que necesita pedirlo a través de la API oficial, que valida permisos, registra el acceso y puede revocar en cualquier momento.

SELECT * FROM products WHERE ...
UPDATE users SET role = 'admin'
DROP TABLE transactions

Base de Datos del Core
Inaccesible directamente · Solo vía API
La única vía válida es api.getProducts(), api.getOrder(id) y similares — funciones acotadas con permisos explícitos.

Contrato Explícito

Permisos Declarados en el manifest.json

Antes de instalar cualquier plugin, el sistema muestra exactamente qué puede y qué no puede hacer. El comercio acepta o rechaza con plena información.

Si el plugin intenta acceder a algo que no declaró, el sandbox lo bloquea y lo registra.

read:productsPuede leer el catálogo de productos para calcular precios.
emit:invoicePuede generar y emitir comprobantes fiscales.
write:usersNo puede modificar usuarios ni roles del sistema.
read:financialsNo puede acceder a reportes financieros completos.
// manifest.json — declaración obligatoria { "name": "FacturAR Pro", "version": "2.1.0", "role": "billing", "author": "dev@facturar.ar", "price": 0, "permissions": [ "read:products", "read:orders", "emit:invoice" ], "sandbox": { "network": true, // solo AFIP endpoints "filesystem": false, "db_direct": false // siempre false } }

Libertad con Límites

Dentro del Sandbox, Todo Esto es Posible

El sandboxing no limita lo que podés construir — limita cómo lo construís. Y eso protege a todos.

Llamadas HTTP externas Conectarse a APIs de terceros (AFIP, MercadoPago, WhatsApp) desde el servidor del plugin.
Leer datos vía API Acceder a productos, órdenes, clientes y configuraciones a través de los endpoints declarados.
Modificar datos autorizados Actualizar precios, emitir facturas, enviar notificaciones — solo lo que el permiso permite.
Renderizar UI propia Mostrar paneles, formularios y reportes dentro del slot visual asignado en la app.
Escuchar eventos del sistema Reaccionar a on_sale_complete, on_stock_import y otros hooks oficiales.
Almacenamiento propio Guardar configuración y estado en su propio espacio aislado, sin tocar tablas del core.

La Última Línea de Defensa

Auditoría Oficial antes de Publicar

El sandbox es técnico, pero hay un segundo control que es humano. Ningún plugin llega al Marketplace sin pasar por revisión del equipo.

No alcanza con que el código no explote. Tiene que tener sentido, ser honesto con lo que declara y no engañar al comercio.

1
Validación estructural automática

El sistema verifica que el ZIP tenga un manifest.json válido, campos obligatorios y sin archivos prohibidos.

2
Revisión de permisos declarados

Se cruza lo que el plugin declara con lo que realmente hace en el código. Si pide read:orders pero nunca lo usa, se marca.

3
Revisión de seguridad humana

Un revisor del equipo lee el código fuente, verifica que no haya intentos de evadir el sandbox, ofuscación sospechosa ni exfiltración de datos.

4
Veredicto y publicación

Aprobado → aparece en el Marketplace. Rechazado → el dev recibe feedback específico para corregir y reenviar.
