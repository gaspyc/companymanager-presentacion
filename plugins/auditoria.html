Ciclo de Publicación

Auditoría
Oficial

Ningún plugin llega al Marketplace sin pasar por revisión. La auditoría es el filtro de confianza que protege a todos los comercios de código malicioso o engañoso.

Dos capas: validación automática primero, revisión humana después.

Validación automática Revisión humana Aprobado / Rechazado Feedback y reenvío

El Proceso

Pipeline de 5 Pasos

Desde que subís el ZIP hasta que aparece en el catálogo, el plugin pasa por cinco etapas bien definidas. Cada una tiene criterios claros de aprobación.

El tiempo promedio de auditoría completa es de 2 a 5 días hábiles.

1
Subida del ZIP

El dev sube el paquete desde "Mis Publicaciones". Estado inicial: Pendiente.

2
Validación estructural automática

El sistema verifica el manifest.json, campos obligatorios, tipos y que no haya archivos binarios prohibidos.

3
Análisis de permisos declarados

Se cruza lo que el plugin declara en permissions con lo que el código realmente llama. Excesos = señal de alerta.

4
Revisión humana de seguridad

Un revisor lee el código fuente. Busca intentos de evadir el sandbox, ofuscación sospechosa y exfiltración de datos.

5
Veredicto final

Aprobado → publicado en el catálogo. Rechazado → feedback con puntos específicos a corregir.

Capa 1 de 2

Validación Automática

El sistema corre estos checks en segundos apenas recibe el ZIP. Si alguno falla, el plugin vuelve al dev con el error específico antes de esperar la revisión humana.

Pasar esta etapa no garantiza aprobación — solo que el paquete está bien formado.

manifest.json presente en raízEl archivo existe y es JSON válido sin errores de sintaxis.
Campos obligatorios completosname, version, role, author, permissions y price están presentes y tienen el tipo correcto.
Versión semántica válidaEl campo version sigue el formato MAJOR.MINOR.PATCH.
Permiso read:financials declarado pero no usadoEl código no llama a ninguna función que requiera este permiso. Se marca para revisión humana.
Archivo .exe encontrado en /srcTipo de archivo no permitido. El ZIP es rechazado automáticamente sin pasar a revisión humana.

Capa 2 de 2

Por Qué Rechaza el Revisor

La revisión humana va más allá de la estructura. Evalúa intención y honestidad. Un plugin bien formado puede igual ser rechazado si el código hace algo diferente a lo que promete.

Estos son los motivos más frecuentes de rechazo en la revisión manual.

Código ofuscado sin justificación

El revisor no puede leer lo que hace el plugin. Sin transparencia no hay aprobación.

Código legible y comentado
Intento de acceso directo a BD

Cualquier import de driver de base de datos o conexión string es rechazo inmediato.

Solo api.getX() y api.setX()
Exfiltración de datos del comercio

Enviar datos de clientes u órdenes a un servidor externo no declarado ni justificado.

Declarar endpoints en sandbox.network
Descripción engañosa

Lo que el plugin promete en la ficha no coincide con lo que realmente hace el código.
