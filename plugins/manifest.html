El Paquete

Estructura del ZIP

El plugin se entrega como un archivo .zip con una estructura específica. El sistema descomprime y valida antes de hacer nada más.

El único archivo obligatorio en la raíz es el manifest.json. El resto es convención recomendada.

mi-plugin-v2.1.0.zip
manifest.json REQUERIDO
index.js Punto de entrada principal
src/ Lógica interna del plugin
ui/ Componentes visuales opcionales
README.md Documentación para el comercio
Los archivos ejecutables fuera de los tipos permitidos (.js, .json, .html, .css, .md) son rechazados automáticamente en la validación estructural.

Campos del manifest.json

Todos los marcados con son obligatorios. El resto es opcional pero recomendado para mejorar la ficha del plugin en el Marketplace.

Rol funcional: billing, payments, pricing, bot, logistics, importer.

Precio mensual en USD. 0 = gratuito. Requiere price_model si es mayor a 0.

access

Visibilidad: public, whitelist o admin. Por defecto: public.

El Corazón del Contrato

Permisos y Configuración de Sandbox

El bloque permissions es lo primero que revisa el auditor. Cada permiso que pedís tiene que justificarse con código que lo use.

El bloque sandbox declara si el plugin necesita red externa o acceso a archivos temporales.

Permisos disponibles

read:productsLee el catálogo
read:ordersLee órdenes y ventas
write:pricesModifica precios vía API
emit:invoiceRegistra comprobantes
hook:priceIntercepta cálculo de precio
emit:notifyEnvía notificaciones

Flags de sandbox

network true Permite llamadas HTTP a APIs externas (AFIP, MP, etc.)
filesystem false Sin acceso al sistema de archivos del servidor
db_direct false Siempre false. No editable. Sin excepciones.

Errores Frecuentes

Por Qué Falla la Validación

El 80% de los rechazos en la etapa automática se deben a los mismos errores. Evitarlos antes de subir el ZIP te ahorra días de espera.

La validación manual puede rechazar por razones adicionales, pero estos son los más comunes en la primera revisión.

Versión mal formateada

Usar "version": "v2" o "2.1" en lugar del formato semántico completo.

Usar "2.1.0" con tres números
Role inválido o inventado

El campo role debe ser uno de los valores del enum oficial. No se aceptan roles personalizados.

billing · payments · pricing · bot · logistics · importer
Permiso declarado que no se usa

Pedir read:orders pero no llamar nunca a api.getOrder() es señal de mala fe o descuido.

Declarar solo lo que el código efectivamente usa
price > 0 sin price_model

Si el plugin es pago, el campo price_model debe declarar "subscription" o "one_time".

Agregar "price_model": "subscription"
